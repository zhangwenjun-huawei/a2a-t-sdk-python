# a2a-t-sdk 开发约束

## 工作流程
1. 绝对不能修改@docs/SPEC.md，这个是项目规格。
2. 对于新任务，细分成小的子任务。每个子任务完成后，提交commit并push到github上。
3. 所有任务、子任务，以及对应的完成情况，写到@docs/TASKS.md里。注意：任务列表写好之后，除非用户让你修改，否则不能修改任务内容，只能更新任务完成情况。
4. **重要：**对于每个子任务，先编写对应功能的UT，然后再实现代码。实现完成后，执行UT，确保语法正确、功能正常。

## 项目结构

## 代码风格

- **语言**: Python 3.14
- **类型注解**: 使用 Python type hints
- **命名**:
  - 类名: `PascalCase`
  - 函数/变量: `snake_case`
  - 常量: `UPPER_SNAKE_CASE`
- **行长度**: 最大 120 字符
- **空行**: 类和函数之间空 2 行，函数之间空 1 行
- 代码要尽量可读，通过合理的架构、方法及变量命名、自注释代码实现、特殊场景/复杂场景的注释来提升可读性。

## 依赖管理

- 所有依赖写在 `requirements.txt`
- 禁止使用未在 requirements.txt 中的包
- 新增依赖需更新 requirements.txt

## 模块设计原则

1. **单一职责**: 每个模块只做一件事
2. **低耦合**: 模块间通过接口通信，不直接操作对方内部数据
3. **可测试**: 核心逻辑单独抽离，便于单元测试

## Git 提交

- 主分支：`main`（受保护，仅通过PR合并）
- 提交前确保代码可运行
- 每次提交后推送到远程个人仓库的develop分支
- 提交信息格式: `类型: description`。**注意**description内容要用英语
  - `feat`: 新功能
  - `fix`: 修复
  - `refactor`: 重构
  - `docs`: 文档
- 每完成一次代码实现/bug修复，在执行下一轮迭代之前，先提交git
