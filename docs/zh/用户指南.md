# 1 Python SDK 用户指南

## 1.1 特性介绍
### 1.1.1 什么是A2A-T？
A2A-T（Agent-to-Agent Telecom）是基于Google A2A协议推出的电信领域多智能体互联协议，专为电信领域复杂协同场景设计。
业界通用智能体互联协议主要关注智能体互联和交互框架，对业务场景和具体交互内容关注不足，导致任务完成的成功率不高。电信领域的业务场景复杂、要求高，运维智能体互联和协同需要有专门的协议进行支撑。A2A-T方案基于A2A协议，重点针对电信领域业务流相关的信息模型、任务协商、协作安全等增强能力进行应用扩展。

### 1.1.2 A2A-T SDK与A2A SDK的关系
 
A2A-T协议是基于Google A2A协议的扩展，针对协议扩展的内容提供了A2A-T SDK，支撑快速构建满用于电信领域复杂协同场景的智能体。A2A-T SDK独立于A2A SDK，通过同时集成A2A-T SDK与A2A SDK可以构建支持A2A-T协议的agent,支持电信领域多agent之间的确定性、高可靠、高效及安全协同。

### 1.1.3 能力介绍

`a2a-t-python` 是面向电信智能体协作场景的 Python SDK，用于在 A2A-T 交互中生成、校验和协商任务提示词。SDK 适合客户端 Agent、服务端 Agent 以及上层编排系统集成使用。

主要能力包括：

- **任务提示词生成**：客户端根据自然语言或结构化输入生成符合 A2A-T 格式的 processed task prompt。
- **服务端提示词校验**：服务端校验客户端下发的 processed task prompt 是否匹配场景、模板和槽位约束。
- **多轮协商**：支持 `information`、`clarification`、`feasibility`、`fulfillment` 四类协商流程。
- **提示词资源管理**：内置场景、槽位、模板和系统提示词资源，支持本地文件资源加载。
- **LLM 适配**：通过 OpenAI-compatible 调用链接入外部大模型。

当前版本为 `0.1.8`，处于 Alpha 阶段，建议用于原型验证、接口联调和能力评估。

## 1.2 应用场景

Python SDK 通常用于以下场景：

1. 客户端 Agent 接收用户意图，生成结构化任务提示词后通过 A2A 协议发送给目标 Agent。
2. 服务端 Agent 接收任务提示词，执行格式、场景和槽位校验，再进入业务处理。
3. 当任务信息不足、任务目标不清晰或能力可行性需要确认时，双方通过 SDK 协商接口推进多轮交互。
4. 样例工程 `a2a-t-samples` 演示了 `subscribe_incident` 场景下的客户端生成、服务端校验和多轮补充协商流程。

## 1.3 环境要求

| 项目 | 要求 |
| --- | --- |
| Python SDK | Python 3.14+ |
| 示例仓 `a2a-t-samples` | 按样例 README 准备 Python 3.14 |
| 依赖管理 | 推荐使用 `uv` |
| LLM | 需要可访问的 OpenAI-compatible 服务及 API Key |
| 操作系统 | Linux、Windows、macOS 均可用于开发和联调 |

> 说明：SDK 自身 `pyproject.toml` 要求 Python 3.12+；当前 `a2a-t-samples` 仓库由于上游依赖约束，README 中要求 Python 3.14，运行样例时以样例仓说明为准。

## 1.4 安装

### 1.4.1 源码方式安装 SDK

```bash
cd {项目路径}/a2a-t-sdk
uv sync --dev
```

执行测试确认环境可用：

```bash
uv run pytest
```

### 1.4.2 准备 SDK 配置

复制环境变量模板：

```bash
cd {项目路径}/a2a-t-sdk
cp package_data/env.example package_data/.env
```

至少需要配置以下内容：

```properties
A2AT_LANGUAGE=zh-CN
A2AT_PROMPT_SOURCE_TYPE=local_file
A2AT_PROMPT_RESOURCE_LOCAL_ROOT_DIR=
A2AT_PROMPT_COMPLIANCE_ENABLED=true
A2AT_PROMPT_COMPLIANCE_GUARDRAIL_PROVIDER=noop
A2AT_LLM_PROVIDER=deepseek
A2AT_LLM_MODEL=deepseek-chat
A2AT_LLM_API_KEY={your_api_key}
A2AT_LLM_BASE_URL=https://api.deepseek.com
A2AT_NEGOTIATION_STATE_STORE_TYPE=in_memory
```

如果 `A2AT_PROMPT_RESOURCE_LOCAL_ROOT_DIR` 为空，SDK 默认使用包内资源。需要使用自定义场景、槽位或模板时，将该配置指向自定义资源根目录。

## 1.5 快速运行示例

Python SDK 的端到端样例位于 `a2a-t-samples` 仓库。

### 1.5.1 安装样例依赖

```bash
cd {项目路径}/a2a-t-samples
uv python pin 3.14
uv sync
```

### 1.5.2 运行 Hello World 示例

```bash
cd {项目路径}/a2a-t-samples/samples/helloworld
cp env.example .env
```

编辑 `.env`，填写：

```properties
A2AT_LLM_API_KEY={your_api_key}
```

启动服务端：

```bash
../../.venv/Scripts/python.exe server_main.py
```

另开终端启动客户端：

```bash
../../.venv/Scripts/python.exe client_main.py
```

预期客户端日志中出现 `Hello world sample completed`，服务端日志中出现 `prompt-check-passed`。

### 1.5.3 运行 Subscribe Incident 示例

```bash
cd {项目路径}/a2a-t-samples/samples/subscribe_incident
cp env.example .env
```

编辑 `.env`，填写：

```properties
A2AT_LLM_API_KEY={your_api_key}
```

如果默认端口被占用，可修改：

```properties
A2AT_SAMPLE_HOST=127.0.0.1
A2AT_SAMPLE_PORT=8888
```

启动服务端：

```bash
../../.venv/Scripts/python.exe server_main.py
```

另开终端启动客户端：

```bash
../../.venv/Scripts/python.exe client_main.py
```

该示例会依次完成：

1. 客户端生成 `subscribe_incident` 任务提示词。
2. 服务端解析并校验任务提示词。
3. 服务端要求补充 `subscription_condition_incident_level`。
4. 客户端补充 incident level 后继续请求。
5. 服务端要求补充 `subscription_condition_incident_name`。
6. 客户端补充 incident name 后继续请求。
7. 服务端返回 `Incident subscription created successfully`。

## 1.6 配置文件速查

| 配置项 | 说明 |
| --- | --- |
| `A2AT_LANGUAGE` | 提示词资源语言，当前常用 `zh-CN` 或 `en-US` |
| `A2AT_PROMPT_SOURCE_TYPE` | 提示词资源来源，当前支持 `local_file` |
| `A2AT_PROMPT_RESOURCE_LOCAL_ROOT_DIR` | 自定义提示词资源根目录 |
| `A2AT_PROMPT_COMPLIANCE_ENABLED` | 是否启用服务端提示词合规校验 |
| `A2AT_PROMPT_COMPLIANCE_GUARDRAIL_PROVIDER` | Guardrail 提供方，当前内置 `noop` |
| `A2AT_LLM_PROVIDER` | LLM 提供方 |
| `A2AT_LLM_MODEL` | 模型名称 |
| `A2AT_LLM_API_KEY` | LLM API Key |
| `A2AT_LLM_BASE_URL` | LLM 服务地址 |
| `A2AT_NEGOTIATION_STATE_STORE_TYPE` | 协商状态存储，当前支持 `in_memory` |

## 1.7 约束与限制

1. 当前版本为 Alpha，接口和资源组织可能随版本演进调整。
2. 内置协商状态存储为内存实现，不提供持久化能力。
3. 提示词资源当前以本地文件为主，不包含注册中心远程资源加载能力。
4. Guardrail 当前内置 `noop` 实现，生产环境需要按安全要求扩展。
5. SDK 负责 A2A-T 提示词生成、校验和协商，不负责 AgentCard 注册、鉴权、服务托管和业务执行。
